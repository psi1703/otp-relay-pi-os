# OTP Relay Portal — Help Docs and RTA Wizard Guide Deployment

This guide documents the markdown-driven Help Docs and RTA Wizard guide pipeline for `psi1703/otp-relay-pi-os`.

The normal portal navigation no longer exposes a standalone **Help** section. The **RTA Wizard** is the primary user-facing guide. Its floating guide content is generated from markdown in:

```bash
docs/help/*.md
```

Screenshots are stored in:

```bash
docs/help/assets/
```

The build script generates the runtime guide data consumed by the portal:

```bash
frontend/help/wizard-guide.json
```

The portal loads that file at:

```text
/help/wizard-guide.json
```

---

## 1. What this guide covers

This document explains:

- how markdown files become RTA Wizard guide content
- where screenshots and generated files live
- how to exclude maintainer-only pages from the wizard
- how the self-hosted GitHub Actions runner rebuilds and deploys help output
- how to troubleshoot common deployment failures

---

## 2. Current operating model

The recommended workflow is:

1. Edit user-facing RTA Wizard guide text in `docs/help/*.md`.
2. Add or replace screenshots in `docs/help/assets/`.
3. Run or let GitHub Actions run `python3 scripts/build_help_docs.py`.
4. The build generates `frontend/help/wizard-guide.json`.
5. The portal loads `/help/wizard-guide.json` at runtime.
6. The workflow syncs generated output into `/opt/otp-relay/frontend/help/`.

Do **not** edit `frontend/help/wizard-guide.json` manually. It is generated output.

---

## 3. Source of truth by content type

| Content type | Source of truth | Generated/live output |
|---|---|---|
| User-facing RTA Wizard guide text | `docs/help/*.md` | `frontend/help/wizard-guide.json` |
| Wizard screenshots/images | `docs/help/assets/` | `frontend/help/assets/` and `/help/assets/` |
| Optional rendered reference pages | `docs/help/*.md` | `frontend/help/rendered/*.html` |
| Help manifest | generated by `scripts/build_help_docs.py` | `frontend/help/manifest.json` |
| Wizard behavior/loading logic | `frontend/app.jsx` | served as `/app.jsx` |
| Wizard layout/design | `frontend/style.css` | served as `/style.css` |

## 4. Folder layout

### Markdown source

```bash
docs/help/
```

This is where normal guide wording should be updated.

### Source images

```bash
docs/help/assets/
```

This is the source of truth for screenshots used by the wizard and optional reference pages.

### Generated output

```bash
frontend/help/
```

This folder is generated by `scripts/build_help_docs.py`. Do not manually maintain content here.

### Live deployed output

```bash
/opt/otp-relay/frontend/help/
```

This folder is served by the portal as `/help/`.

## 5. Wizard frontmatter controls

Map a markdown page to one or more wizard steps:

```yaml
---
title: Reset RTA Password
section: Access
order: 2
slug: reset-rta-password
wizard_steps: [password_reset]
---
```

Exclude a page from the wizard while keeping it as an optional rendered reference page:

```yaml
---
wizard: false
---
```

This is important for maintainer-only pages such as:

```bash
docs/help/00-overview.md
```

Portal users should not see documentation that explains the docs pipeline itself.

If no `wizard_steps` value is present, `scripts/build_help_docs.py` uses its default file-to-step mapping.

## 6. Referencing screenshots

Put screenshots in:

```bash
docs/help/assets/
```

Reference them from markdown like this:

```md
![VPN request form](assets/vpn-request-form-details.png)
```

The build script rewrites the rendered path to:

```text
/help/assets/vpn-request-form-details.png
```

## 7. Build script behavior

Build script:

```bash
scripts/build_help_docs.py
```

It does the following:

1. Reads markdown from `docs/help/`.
2. Copies assets from `docs/help/assets/` to `frontend/help/assets/`.
3. Converts markdown into optional rendered HTML files under `frontend/help/rendered/`.
4. Generates `frontend/help/manifest.json`.
5. Generates `frontend/help/wizard-guide.json`.
6. Respects `wizard: false` for pages that must not appear in the RTA Wizard overlay.

Source flow:

```text
docs/help/*.md + docs/help/assets/*
        ↓
scripts/build_help_docs.py
        ↓
frontend/help/wizard-guide.json
frontend/help/rendered/*.html
frontend/help/assets/*
frontend/help/manifest.json
        ↓
workflow sync
        ↓
/opt/otp-relay/frontend/help/
```

## 8. Recommended workflow file

Workflow file:

```bash
.github/workflows/deploy-help-docs.yml
```

The workflow can keep that filename. A clear workflow name is:

```yaml
name: Deploy Help Docs and Wizard Guide
```

Recommended trigger paths:

```yaml
on:
  push:
    branches: [ main ]
    paths:
      - "docs/help/**"
      - "scripts/build_help_docs.py"
      - ".github/workflows/deploy-help-docs.yml"
  workflow_dispatch:
```

The build step should run:

```bash
python3 scripts/build_help_docs.py
```

The sync step should deploy generated help output:

```bash
rsync -rltvz --delete --no-group --no-owner frontend/help/ /opt/otp-relay/frontend/help/
```

## 9. Day-to-day usage

### Update guide wording shown to users

Edit the relevant markdown topic:

```bash
nano docs/help/02-reset-rta-password.md
```

Then rebuild and deploy:

```bash
python3 scripts/build_help_docs.py
python3 scripts/deploy_portal_ui.py
sudo systemctl restart otp-relay
```

### Add or replace a screenshot

1. Put the image in:

   ```bash
   docs/help/assets/
   ```

2. Reference it from markdown:

   ```md
   ![Description](assets/example.png)
   ```

3. Rebuild:

   ```bash
   python3 scripts/build_help_docs.py
   ```

### Keep a markdown page out of the wizard

Add this frontmatter:

```yaml
---
wizard: false
---
```

Use this for maintainer-only pages or pages that should remain reference-only.

## 10. Manual verification commands on the Pi

Check generated files:

```bash
ls -R frontend/help
```

Check the wizard JSON endpoint:

```bash
curl -s -o /dev/null -w "wizard=%{http_code}\n" http://127.0.0.1:8000/help/wizard-guide.json
```

Check a screenshot endpoint:

```bash
curl -s -o /dev/null -w "asset=%{http_code}\n" http://127.0.0.1:8000/help/assets/<filename>
```

Inspect generated wizard data:

```bash
python3 -m json.tool frontend/help/wizard-guide.json | less
```

Search for accidentally exposed maintainer text:

```bash
grep -R "generated from markdown" frontend/help/wizard-guide.json
grep -R "source of truth" frontend/help/wizard-guide.json
```

Those commands should not return user-facing wizard entries unless that wording was intentionally placed in a user-facing topic.

## 11. Troubleshooting

### Maintainer documentation appears in the RTA Wizard

Cause: a maintainer-only markdown page is mapped to a wizard step.

Fix:

1. Add `wizard: false` to the page frontmatter.
2. Make sure `scripts/build_help_docs.py` does not map that filename in `DEFAULT_WIZARD_STEP_MAP`.
3. Rebuild:

   ```bash
   python3 scripts/build_help_docs.py
   ```

4. Deploy and restart:

   ```bash
   python3 scripts/deploy_portal_ui.py
   sudo systemctl restart otp-relay
   ```

### Markdown changed but wizard text did not change

Check:

- `frontend/help/wizard-guide.json` was regenerated
- the portal can fetch `/help/wizard-guide.json`
- the browser is not showing cached content
- the markdown page is not marked `wizard: false`
- the markdown page maps to the expected wizard step

### Screenshots did not update

Check:

- the image exists in `docs/help/assets/`
- the generated copy exists in `frontend/help/assets/`
- the live copy exists in `/opt/otp-relay/frontend/help/assets/`
- the markdown references `assets/<filename>`
- the browser cache was refreshed

### PNG screenshots disappeared

Cause: `rsync --delete` removed files that were not present in the generated source tree.

Fix:

- commit all source screenshots under `docs/help/assets/`
- do not manually store screenshots only in `frontend/help/assets/` or `/opt/otp-relay/frontend/help/assets/`

### Runner shows Offline in GitHub

Check the runner service:

```bash
sudo systemctl status actions.runner.*.service
sudo journalctl -u actions.runner.*.service -n 50
```

Restart if needed:

```bash
sudo systemctl restart actions.runner.*.service
```

## 12. Operational rules

- `docs/help/*.md` is the source of truth for RTA Wizard guide text.
- `docs/help/assets/` is the source of truth for wizard and reference screenshots.
- `frontend/help/` is generated output; do not manually author content there.
- `/opt/otp-relay/frontend/help/` is live deployed output; do not manually author content there.
- Use `wizard: false` for maintainer-only pages.
- Use `wizard_steps` only when a page must map to specific wizard steps.
- Keep screenshot filenames stable once referenced from markdown.

## 13. Summary

This setup supports:

- RTA Wizard guide copy maintained in markdown
- screenshots maintained in `docs/help/assets/`
- generated wizard JSON consumed by the portal
- optional rendered Help reference pages
- automatic build and deployment on the Raspberry Pi

Key principle:

**Edit markdown and assets → build help docs → portal loads generated wizard-guide JSON.**
