# OTP Relay Portal — Help Docs and RTA Wizard Deployment Guide

This guide documents the Help Docs / RTA Wizard guide automation for `otp-relay-pi-os`.

The normal portal navigation no longer exposes a standalone **Help** section. The **RTA Wizard** is now the primary user-facing guide surface.

The RTA Wizard floating guide is **markdown-driven**:

- edit guide text in `docs/help/*.md`
- put screenshots in `docs/help/assets/`
- run `python3 scripts/build_help_docs.py`
- deploy the portal UI

The build generates:

```text
frontend/help/wizard-guide.json
```

The portal loads this file at runtime from:

```text
/help/wizard-guide.json
```

So normal guide text updates do **not** require editing the large `frontend/app.jsx` file.

---

## 1. What this guide covers

This document explains:

- how the RTA Wizard guide and Help Docs pipeline relate to each other
- where user-facing guide text, screenshots, generated files, and live files live
- how to map markdown files into wizard steps
- how the self-hosted GitHub Actions runner rebuilds and deploys guide output
- how to add or replace screenshots used by the wizard overlay
- how to maintain optional generated Help reference pages
- how to troubleshoot common deployment failures

## 2. Current operating model

The current recommended workflow is:

1. Edit user-facing guide content in `docs/help/*.md`.
2. Add or replace screenshots in `docs/help/assets/`.
3. Use markdown frontmatter to map pages to RTA Wizard steps when needed.
4. Run the build script:

   ```bash
   python3 scripts/build_help_docs.py
   ```

5. Deploy the portal UI:

   ```bash
   python3 scripts/deploy_portal_ui.py
   sudo systemctl restart otp-relay
   ```

6. The portal loads the generated wizard guide from `/help/wizard-guide.json`.

The standalone Help section is not part of the normal user navigation anymore. The RTA Wizard is the main user-facing guide surface.

## 3. Source of truth by content type

| Content type | Source of truth | Notes |
|---|---|---|
| User-facing RTA Wizard guide text | `docs/help/*.md` | Primary content users see in the RTA Wizard floating guide |
| Wizard screenshot source files | `docs/help/assets/` | Served after build as `/help/assets/<filename>` |
| Wizard guide generated data | `frontend/help/wizard-guide.json` | Generated by `scripts/build_help_docs.py`; do not hand-edit |
| Optional Help reference markdown | `docs/help/*.md` | Same markdown can also generate fallback/reference HTML pages |
| Optional rendered Help output | `frontend/help/rendered/*.html` | Generated; do not hand-edit |
| Help manifest | `frontend/help/manifest.json` | Generated; used for optional rendered reference pages |
| Generated Help assets | `frontend/help/assets/` | Generated from `docs/help/assets/`; do not hand-edit |
| Live deployed Help output | `/opt/otp-relay/frontend/help/` | Live `/help/` folder served by the portal |
| Wizard UI behavior | `frontend/app.jsx` | Edit only for behavior, loading logic, links, and step structure |
| Wizard visual design | `frontend/style.css` | Edit only for layout, floating guide styling, and responsive behavior |

## 4. Folder layout

### RTA Wizard UI behavior

```bash
frontend/app.jsx
```

This file contains the React/JS UI logic for the RTA Wizard and floating guide overlay. Do not edit this file for routine guide wording changes.

Edit this file only when changing guide behavior, for example:

- how the floating window opens or closes
- draggable-window behavior
- guide tab navigation behavior
- runtime loading of `/help/wizard-guide.json`
- wizard step structure or IDs
- external link behavior

### RTA Wizard visual design

```bash
frontend/style.css
```

This file contains the visual styling for the portal, wizard, and floating guide. Edit this only for design/layout changes.

### Source markdown guide pages

```bash
docs/help/
```

These markdown files are the source of truth for normal RTA Wizard guide wording.

They also remain useful as optional rendered fallback/reference pages.

### Source images

```bash
docs/help/assets/
```

This is the source of truth for screenshots used by both the RTA Wizard floating guide and optional generated Help pages.

### Generated output

```bash
frontend/help/
```

This folder is generated by `scripts/build_help_docs.py`. Do not manually maintain content here.

Expected generated files include:

```text
frontend/help/wizard-guide.json
frontend/help/manifest.json
frontend/help/rendered/*.html
frontend/help/assets/*
```

### Live deployed output

```bash
/opt/otp-relay/frontend/help/
```

This folder is served by the portal as `/help/`.

The RTA Wizard floating guide consumes:

```text
/help/wizard-guide.json
/help/assets/<filename>
```

## 5. Markdown frontmatter controls

Each markdown page can include YAML frontmatter at the top.

Example:

```yaml
---
title: Reset RTA Password
section: Access
order: 2
slug: reset-rta-password
wizard_steps: [password_reset]
---
```

### Map a page to one or more wizard steps

Use `wizard_steps`:

```yaml
wizard_steps: [password_reset]
```

or:

```yaml
wizard_steps:
  - form
  - account_creation
  - save_iits
```

### Exclude a page from the wizard overlay

Use:

```yaml
wizard: false
```

This keeps the page available for optional rendered reference docs while excluding it from the RTA Wizard floating guide.

### Default mapping

If no `wizard_steps` value is present, `scripts/build_help_docs.py` uses its default file-to-step mapping.

Use explicit `wizard_steps` frontmatter when you want predictable behavior.

Use these IDs when mapping markdown pages to wizard steps:

| Wizard step ID | Meaning |

| `vpn_request` | Request VPN / RDP / PAM / SFTP access |

## 7. Screenshots and images

All screenshots used by the RTA Wizard floating guide or optional Help reference pages must be added to:

```bash
docs/help/assets/
```

Do not manually maintain screenshots in:

```bash
frontend/help/assets/
/opt/otp-relay/frontend/help/assets/
```

Those folders are generated/deployed outputs and may be overwritten by the next build.

### Referencing images from markdown

In `docs/help/*.md`, reference screenshots like this:

```md
![VPN request form](assets/vpn-request-form-details.png)
```

During the build, markdown image paths are converted into:

```text
/help/assets/vpn-request-form-details.png
```

for the portal.

### Referencing images from app.jsx

Only do this for non-routine UI/behavior changes. Routine wizard guide screenshots should be referenced from markdown.

If needed, app-level paths should use:

```js
'/help/assets/example.png'
```

## 8. Build script behavior

Build script:

```bash
scripts/build_help_docs.py
```

It does the following:

1. Reads markdown guide/reference pages from `docs/help/`.
2. Reads frontmatter such as `title`, `section`, `order`, `slug`, `wizard`, and `wizard_steps`.
3. Deletes and recreates `frontend/help/assets/`.
4. Copies all files from `docs/help/assets/` into `frontend/help/assets/`.
5. Converts markdown into optional rendered HTML files in `frontend/help/rendered/`.
6. Generates `frontend/help/manifest.json`.
7. Generates `frontend/help/wizard-guide.json` for the RTA Wizard floating guide.
8. Rewrites markdown image paths from `assets/...` to `/help/assets/...`.

Source flow:

```text
docs/help/*.md + docs/help/assets/*
        ↓
scripts/build_help_docs.py
        ↓
frontend/help/wizard-guide.json
frontend/help/rendered/*.html
frontend/help/assets/*
frontend/help/manifest.json
        ↓
portal deploy
        ↓
/opt/otp-relay/frontend/help/
        ↓
portal runtime loads /help/wizard-guide.json
```

## 9. Recommended workflow file

Workflow file:

```bash
.github/workflows/deploy-help-docs.yml
```

Recommended workflow name and job name:

```yaml
name: Deploy Help Docs and Wizard Guide

on:
  push:
    branches: [ main ]
    paths:
      - "docs/help/**"
      - "scripts/build_help_docs.py"
      - "frontend/app.jsx"
      - "frontend/style.css"
      - ".github/workflows/deploy-help-docs.yml"
  workflow_dispatch:

jobs:
  deploy-help-docs-and-wizard-guide:
    runs-on: [self-hosted, Linux, ARM64]

    steps:
      - name: Checkout repo
        uses: actions/checkout@v5

      - name: Use system Python
        run: python3 --version

      - name: Install build dependencies
        run: |
          python3 -m pip install --break-system-packages markdown pyyaml

      - name: Build help docs and wizard guide
        run: |
          python3 scripts/build_help_docs.py

      - name: Confirm generated files
        run: |
          ls -R frontend/help
          cat frontend/help/manifest.json
          test -f frontend/help/wizard-guide.json

      - name: Sync built help output to live portal
        run: |
          rsync -rltvz --delete --no-group --no-owner frontend/help/ /opt/otp-relay/frontend/help/
```

The workflow can keep the filename `deploy-help-docs.yml`.

---

## 10. Why `rsync --delete` is used

The deployment uses:

```bash
rsync -rltvz --delete --no-group --no-owner frontend/help/ /opt/otp-relay/frontend/help/
```

This makes the live `/help/` output match the generated output exactly.

That is good because:

- deleted optional markdown pages disappear from generated reference output
- renamed assets do not leave old junk behind
- stale `wizard-guide.json` output is replaced
- the live folder stays clean
- the RTA Wizard floating guide only sees current screenshots from `/help/assets/`

Important warning: if a file is only placed manually in the live folder and not generated from the repo, it will be deleted on the next deployment.

## 11. Day-to-day usage

### Update user-facing RTA Wizard guide copy

Edit one or more files in:

```bash
docs/help/*.md
```

Then rebuild:

```bash
python3 scripts/build_help_docs.py
```

Deploy:

```bash
python3 scripts/deploy_portal_ui.py
sudo systemctl restart otp-relay
```

### Add or replace a wizard screenshot

1. Put the PNG in:

   ```bash
   docs/help/assets/
   ```

2. Reference it from a markdown page:

   ```md
   ![Example screenshot](assets/example.png)
   ```

3. Rebuild locally if you want to verify before pushing:

   ```bash
   python3 scripts/build_help_docs.py
   ```

4. Commit and push.

### Add or update an optional Help reference page

1. Create or edit a markdown file in `docs/help/`.
2. Keep YAML frontmatter at the top:

   ```md
   ---
   title: Example Page
   section: Example
   order: 20
   slug: example-page
   wizard: false
   ---
   ```

3. Reference images using `assets/<filename>`.
4. Commit and push.

### Change floating guide behavior

Edit:

```bash
frontend/app.jsx
```

Only use this for behavior or structure changes, not routine guide wording.

### Change floating guide layout/design

Edit:

```bash
frontend/style.css
```

Only use this for visual/layout changes.

## 13. Permissions required for deployment

The runner user, normally `initbox`, must be able to write into:

```bash
/opt/otp-relay/frontend/help/
```

This is normally handled by `install.sh`. If permissions need to be restored manually:

```bash
sudo chown -R initbox:initbox /opt/otp-relay/frontend/help
find /opt/otp-relay/frontend/help -type d -exec chmod 755 {} \;
find /opt/otp-relay/frontend/help -type f -exec chmod 644 {} \;
```

If the workflow fails with `Permission denied`, `Operation not permitted`, or `rsync error code 23`, run the commands above and re-trigger the workflow.

## 14. Troubleshooting

### Workflow ran but wizard screenshots did not change

Check:

- the screenshot is committed under `docs/help/assets/`
- the markdown page references the screenshot as `assets/<filename>`
- `scripts/build_help_docs.py` copied it into `frontend/help/assets/`
- the workflow sync step ran successfully
- `/opt/otp-relay/frontend/help/assets/<filename>` exists
- `/help/assets/<filename>` returns HTTP 200
- the browser is not showing cached content

### PNG screenshots disappeared

Cause: `rsync --delete` removed files that were not present in the generated source tree.

Fix:

- make sure all screenshots are committed in `docs/help/assets/`
- do not treat `frontend/help/assets/` or `/opt/otp-relay/frontend/help/assets/` as hand-maintained storage

### Markdown changed but wizard text did not change

Check:

- the markdown file is under `docs/help/`
- the page is not marked `wizard: false`
- `wizard_steps` maps to the expected wizard step ID
- `python3 scripts/build_help_docs.py` was run
- `frontend/help/wizard-guide.json` contains the new text
- `/opt/otp-relay/frontend/help/wizard-guide.json` contains the new text after deploy
- the browser is not showing cached content

### Wizard guide shows fallback/static content

This usually means the portal could not load:

```text
/help/wizard-guide.json
```

Check:

```bash
curl -s -o /dev/null -w "wizard=%{http_code}\n" http://127.0.0.1:8000/help/wizard-guide.json
```

If the status is not `200`, rebuild and deploy:

```bash
python3 scripts/build_help_docs.py
python3 scripts/deploy_portal_ui.py
sudo systemctl restart otp-relay
```

### Runner shows Offline in GitHub

Check the runner service:

```bash
sudo systemctl status actions.runner.*.service
sudo journalctl -u actions.runner.*.service -n 50
```

Restart if needed:

```bash
sudo systemctl restart actions.runner.*.service
```

## 15. Operational rules

- `docs/help/*.md` is the source of truth for user-facing RTA Wizard guide text.
- `docs/help/assets/` is the source of truth for wizard and reference screenshots.
- `frontend/help/wizard-guide.json` is generated output for the RTA Wizard floating guide; do not hand-edit it.
- `frontend/help/manifest.json` and `frontend/help/rendered/*.html` are generated output for optional reference pages; do not hand-edit them.
- `frontend/help/assets/` is generated output; do not manually author content there.
- `/opt/otp-relay/frontend/help/` is live deployed output; do not manually author content there.
- Edit `frontend/app.jsx` only for guide behavior, loading logic, or wizard structure changes.
- Edit `frontend/style.css` only for visual/layout changes.
- Store all source images in `docs/help/assets/`.
- Keep screenshot filenames stable once referenced from markdown.

## 16. Summary

This setup supports:

- RTA Wizard guide copy maintained in `docs/help/*.md`
- screenshot assets maintained in `docs/help/assets/`
- automatic generation of `frontend/help/wizard-guide.json`
- optional rendered Help reference pages maintained from the same markdown source
- automatic Help Docs / wizard guide build on the Raspberry Pi
- automatic live deployment to `/opt/otp-relay/frontend/help/`
- a clean separation between source files, generated output, live deployed output, UI behavior, and visual design

Key principle:

**Edit guide text in `docs/help/*.md` and images in `docs/help/assets/` → run `scripts/build_help_docs.py` → portal loads `/help/wizard-guide.json`.**
